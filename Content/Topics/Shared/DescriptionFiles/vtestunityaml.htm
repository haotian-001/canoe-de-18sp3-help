vtestunit.yaml

 

Valid for:  CANoe DE  •  CANoe4SW DE

In a vtestunit.yaml file a test unit is described textually. Here vtestunit.yaml is not the filename, but a file extension, e.g. TestUnitName.vtestunit.yaml.

The format used is YAML. You can edit this format with any text editor, but a text editor with YAML support is recommended.

Note about the Migration of Test Modules

(available for CANoe DE)

The document TestModule_vtestunit-yaml_MigrationManual.pdf describes how you can migrate test modules (CAPL, C#, XML) for the execution with canoe4sw-se bzw. canoe-se to test units in the vtestunit.yaml format.

Structure of vtestunit.yaml

You can find a complete scheme structure here: vtestunit.yaml schema

The specification of version is mandatory; all other elements of the structure are optional.

version: 2.0.0
includes:
...
test-unit-information:
...
test-unit-implementation:
...

Test Unit Information

The test-unit-information element lists general, optional information about the test unit.

This information is available in the test report after execution. The title, version and description of the test unit can be specified. Furthermore, it is possible to create information about the test developers and additional freely definable key-value pairs.

test-unit-information:
  caption: This is a caption
  description: This is a description
  version: v2.0
  test-designers:
    - name: Max
      department: PND123
    - name: Lisa
      department: PND456
  additional-information:
    - name: Software Under Test
      CustomerName: X012
      SpareNumber: 0 145 Y03
      SRSBaseline: 4.1

Test Unit Implementation

The test-unit-implementation element lists all artifacts that belong to the test implementation. It is a list of key-value pairs, each of which must match one of the following key definitions (oneOf).

These are relative or absolute paths to files in source form, but also compiled libraries. Relative paths are resolved relative to the *.vtestunit.yaml file. Each artifact is defined as a key-value pair.

  • source-file-path

    Path to a CAPL file (*.can), encrypted CAPL file (*.canencr), .NET file (*.cs), Python file (*.py) or vtesttree.yaml file.

  • capl-library-path

    Path to a CAPL library (*.dll).

  • python-package-path

    Path to a Python package.

  • dotnet-assembly-path

    Path to a .NET assembly file.

  • dotnet-framework-assembly-name

    .NET framework assembly name.

  • modeling-library-path

    Path to a modeling library (*.vmodule).

test-unit-implementation:
#test tree
  - source-file-path: MyTestTree.vtesttree.yaml
# feature tests
  - source-file-path: MyFeatureTest.can
  - source-file-path: MyEncCaplTest.canencr
  - source-file-path: MyDotnetTest.cs
  - source-file-path: MyPythonTest.py
# diagnostic tests
  - source-file-path: D:/dev/MyDiagTest.can
# Libs
  - capl-library-path: custom.dll
  - capl-library-path: custom2.dll
  - python-package-path: path/to/my/python/package
  - dotnet-assembly-path: path/to/my.dll
  - modeling-library-path: path/to/NodeLayer2.vmodule

Tool Recommendation

Visual Studio Code with YAML Extension from Red Hat.

The vtestunit.yaml schema should be obtained automatically from the schema store. If this is not possible, for example because there is no internet connection or there are proxy problems, you can also configure the schema manually in the Visual Studio Code settings.

{
  "yaml.schemas": {
    "file://<Tool_Installation_Directory>/Exec64/Schemas/vtestunit.yaml-schemas/vtestunit-schema.json"
: ["*.vtestunit.yaml"]
  },
}
