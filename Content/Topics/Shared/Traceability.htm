Traceability

 

Valid for:  CANoe DE  •  CANoe:lite DE  •  CANoe4SW DE  •  CANoe4SW:lite DE

Introduction

You can trace externally defined requirements and test descriptions in the test implementation and in the test report (traceability). This is done via an exchange file in a predefined open XML format. This means that any REQM/TDM system can be used.

The information exported from the REQM/TDM system is referenced in the form of a file (*.vti-tso) in vtestunit.yaml and can then be linked to test cases via annotations in the programming languages.

The linked elements can be requirements or test descriptions. Because these elements are used to achieve traceability, they are designated as trace items.

For each trace item linked to a test case, the test report contains a corresponding reference at the test case. This enables traceability from the test case result back to the requirement or test description.

Workflows

In general, the following workflows for coupling REQM/TDM systems are supported. Depending on the specific REQM/TDM system and its data model, the individual steps and involved artifacts may differ.

Workflow Based on Requirements

With this approach, the requirements of the REQM/TDM system are exported as trace items. The test design is done directly on the basis of the requirements or trace items to be tested.

When the test unit is executed, a test report is created that can be used for the subsequent feedback of the results into the REQM/TDM system. For each test case, a corresponding test case is automatically created or updated in the REQM/TDM system if it already exists. The links between the test cases and the trace items are also transferred, and traceability to the requirements is established in the REQM/TDM system. The test result is assigned to the test cases in the REQM/TDM system using the unique test case IDs.

Workflow Based on Test Cases

In contrast to the requirement-based workflow, the test cases in this approach are already available in the REQM/TDM system, i.e. they have been specified in advance. These test case descriptions are exported as trace items in order to implement the test cases based on them. Here, the trace items and the test cases are linked in a one-to-one relationship, i.e. for each test case description in the REQM/TDM system there is exactly one test case implementation and vice versa.

When returning the test report, the test results are mapped using the referenced trace item, which refers directly to the test case in the REQM/TDM system.

Supported REQM/TDM Systems

The following external REQM/TDM systems are supported out of the box:

  • Vector PREEvision TDM (integrated, no add-on needed)
  • IBM Rational DOORS
  • IBM Rational DOORS NG, IBM Rational Quality Manager / IBM Engineering Test Management
  • PTC Integrity / Windchill
  • Siemens Polarion ALM
  • Jama Connect
  • Intland codeBeamer ALM

The tool-specific add-ons can be downloaded for free on the Vector homepage http://vector.com/vteststudio-addons/. Each add-on contains an additional manual which describes the functionality of the respective tool coupling in more detail.

Any other REQM/TDM systems can be connected by their own plugins. The descriptions of the required XML-based exchange formats are available as XSD files in the <Installationsverzeichnis>\Exec64\Schemas\traceability-schemas folder.

If required, Vector will support additional REQM/TDM systems by default in the future.

Linking of Trace Items to Test Cases

After exporting trace items from a REQM/TDM system, they are available as a list in an XML format (*.vti-tso). Each trace item has a UniqueID. This can be used to link a trace item to a test case implementation.

When linking test cases to trace items, any n:m relationships are possible, i.e. a test case can also be linked to several trace items. In a workflow based on test cases, however, a 1:1 relationship between a test case and a trace item is expected.

Example of a *.vti-tso from Polarion:

CAPL

A trace item is linked to a test case implementation in CAPL via the annotation traceitem. The UniqueID of the trace item is transferred as a string.

Example

[traceitem("CLS-511")]
export testcase MyCaplTestCase()
{

}

C#

A trace item is linked to a test case implementation in C# via the Vector.CANoe.TFS.TraceItem attribute. The UniqueID of the trace item is transferred as a string.

Example

  [Export][TestCase]
  [Vector.CANoe.TFS.TraceItem("CLS-511")]
  public static void MyCsTestCase()
  {

  }

Python

A trace item is linked to a test case implementation in Python via the decorator vector.canoe.tfs.trace_item. The UniqueID of the trace item is passed as a string. The trace_item decorator must be specified before the test_case decorator.

Example

@vector.canoe.tfs.export
@vector.canoe.tfs.trace_item("CLS-511")
@vector.canoe.tfs.test_case
def MyPythonTestCase():

CANoe Execution Adapter

The Vector CANoe Execution Adapter can be used for automatic test case selection and execution in your CANoe DE product.

An XML-based planning list containing the test cases to be executed serves as the source.

  • Execution Plan (*.vexecplan)

    This format contains a flat list of test cases. For example, execution planning from external REQM/TDM systems can be realized using this format.

    Export source: REQM/TDM System, CANoe Test Report Viewer

    Further export sources can be realized by own implementations. The XML-based exchange format is described in an XSD file, see <Installationsverzeichnis>\Exec64\Schemas\traceability-schemas\ExecutionPlan.xsd.

The CANoe Execution Adapter is available as GUI application and as command line tool. It can also be downloaded as free add-on from the Vector homepage.
