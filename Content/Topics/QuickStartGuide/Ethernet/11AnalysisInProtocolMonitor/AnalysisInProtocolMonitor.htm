Analyse im Protocol Monitor

 

Valid for:  CANoe DE

The Protocol Monitor observes data traffic in the network and interprets a variety of protocols supported by CANoe DE. The connections between endpoints are displayed graphically and in tabular form. In a hierarchical tree view, all endpoints are broken down by their affiliation to network nodes or to networks.

In the created configuration, you can use it to check the SOME/IP communication between the simulated nodes, for example.

Add a Protocol Monitor

As a lot of data is displayed in the Protocol Monitor, you should open it on a separate desktop.

Create with a new desktop with the name Protocol Monitor.

Activate the Protocol Monitor via the Analysis ribbon tab.

Like all analysis windows, the Protocol Monitor is only filled with data when the measurement is started. It evaluates all events during the measurement and displays the current status of your network; there is no data history as in the Trace Window, for example.

The Protocol Monitor displays information in different views. As the amount of data from SOME/IP services is usually very large, you should limit the display of the data with filters (see below).

Endpoint View

In the endpoint view on the Communication View page, all Ethernet endpoints that communicate with each other during the measurement are displayed. These are all four simulation nodes and one Multicast node.

The SOME/IP Service Discovery uses a multicast IP address to process information that is sent to all interested nodes in the network. All multicast IP addresses are displayed in the Multicast node, which is not a real node.

Click to collapse the node view.

Click on a node to hide all other nodes in the Endpoint View and also on the right in the Diagram View.

Click <ESC> to cancel the selection again.

The different colors in the extended endpoint view represent the different OSI layers. Gray stands for the data link layer (MAC addresses), green for the network layer (IP addresses), etc.

Diagram View

In the Diagram View on the Communication View page, the nodes and their OSI layers are displayed graphically.

The Diagram View shows the connections between the different nodes, symbolized by arrows between sender and receiver.

In our tour we observe SOME/IP packets. ADAS provides a service on VLAN 3. We expect to see two arrows in the diagram view. An arrow from ADAS to a multicast address for the IGMP protocol (to announce that ADAS is using SOME/IP). Another arrow for the SOME/IP service offer sent to another multicast address.

Collapse the view for a better overview as follows:

Hide the IC, CAMF and VGW nodes in the Endpoint View via the respective context menu. The hidden nodes are marked with .

Select Ethernet1::3 at the top left of the diagram view to focus only on VLAN 3.

Only ADAS and Multicast as well as 2 connections are now displayed in the diagram view.

The green arrow on the network layer is the IGMP packet, the blue arrow is the SOME/IP SD Service offer. Click on one to select it. Then a line will be highlighted in the Connection View and the details of this connection will be displayed in the Detail View.

Text Search in the Diagram View

SOME/IP is used to dynamically add and remove connections between information providers and consumers.

If you want to know during the measurement whether a control device is currently subscribed to a specific service from a provider or not, you can use the text search to display the relevant information for checking connections.

Step by Step Procedure

  1. Show all nodes again.

  2. Select All Networks at the top left.

  3. Start the text search with in the toolbar of the Protocol Monitor.

  4. Enter sif_1 in the search field.

    The text search will now display all services with an ID beginning with 1, in our example the services with the IDs 1000, 1001, 1002 and 1003.

  5. If you only want to see the subscriptions for these services, click on next to the search input field to open the advanced search.

    You can compile your search here using various predefined parameters.

  6. Select the first parameter Subscriptions from the selection list by double-clicking on it.

  7. Select the operator ~ directly after the first parameter. The tilde character (ASCII no. 126) searches for an occurrence of the character string.

  8. Enter sif_1 directly after the operator.

    This search returns fewer hits.

    Double-click on any line to obtain more information in the Detail view. It may be interesting to note that service 1000 is offered on VLAN 1 and 3. Logically, these are different networks. In addition to the four operators mentioned above, it is also possible to use logical operators AND/OR for the text search. An example would be the search for: Subscriptions ~ sif_1 AND Subscriptions ~ sif_2. This means that we are searching for endpoints that are subscribed to at least two services. One service ID must start with 1 and the other with 2.

Connection View

Connections between endpoints are displayed in the Connection View on the Communication View page. The time stamps come from the Ethernet packets that belong to the respective connection. Connections can be unidirectional, e.g. an AUTOSAR PDU sent via UDP. They can also be bidirectional, e.g. SOME/IP SD for an event group (find service, subscribe event group, etc. / offer service, subscribe acknowledge, etc.). The direction is listed in a column. The status shows whether the connection currently exists or was only seen at an earlier point in time during the measurement.

The selection of the Ethernet1::3 network also has an effect here. To further reduce the number of lines for reasons of clarity, you can define a pass-through filter for AUTOSAR PDU connections here.

Stop and Pass Filter

Step by Step Procedure

  1. To do this, open the splitter on the left-hand side of the Connection View.

  2. Click on in the toolbar to add a Pass Filter.

  3. Select the new Pass Filter and open the dialog for defining a property condition by clicking in the toolbar.

  4. Enter the following:

    Property: PDUs

    Operator: ~ (contains the value in the PDU name)

    Value: Remains empty so that the filter matches every PDU.

  5. Click on to accept the condition.

  6. Close the dialog with .

  7. Activate the filter.

    The filter affects all views on the Communication View page. The Connection View now only shows that a PDU has been sent from VGW to ADAS. Click on the line to see the details in the Detail View on the left in the Protocol Monitor.

  8. Now select All Networks in the diagram view at the top left.

    A second line is now displayed in the connection view. On VLAN 2, the VGW sends PDUs to the IC node.

Trace Window Filter

You can create filters for the Trace Window for endpoints, connections and networks.

After analyzing that PDUs are sent from VGW to IC, these PDUs should be displayed in chronological order in the Trace Window.

Step by Step Procedure

  1. First deactivate all analysis filters in the Trace Window.

  2. To do this, switch to the Trace desktop and open the analysis filters with in the toolbar in the Trace Window.

  3. Deactivate the Filter Group 0 here or deactivate all analysis filters with in the toolbar.

  4. Switch back to the Protocol Monitor desktop.

  5. Select the second line (VLAN ID 2) in the Connection View.

  6. Use the context menu command Create Filter for Trace Window|Trace to open the Trace filter dialog to create a filter for the PDU connection from VGW to IC on VLAN 2 for the Trace Window Trace.

  7. Click on [Create] to create the Pass Filter for the Trace Window.

  8. Switch to the Trace desktop and open the analysis filter in the Trace Window with in the toolbar.

    Here you can see the filter you have just created. It consists of several filter groups.

  9. Start the measurement.

    You will see Ethernet packets containing the PDUs and the PDUs themselves.

Detail View

The Detail View shows further information on the currently selected object in one of the views mentioned above.
