Debugger: Debugging

 

Valid for:  CANoe DE  •  CANoe:lite DE  •  CANoe4SW DE  •  CANoe4SW:lite DE

Debug Mode

Prior to starting a measurement, debug mode can be activated for as many programs as necessary. Each program, test modulevalid for CANoe DE, CANoe:lite DE or test configuration is displayed in a separate Debugger Window. In simulated mode, debug mode remains active for all supported program types. For a measurement with real hardware, it is only activated for test modules and test configurations because the simulation would have to be stopped for network nodes.valid for CANoe DE, CANoe:lite DE

If debug mode is deactivated, no variable values are displayed for the node, the test modulevalid for CANoe DE, CANoe:lite DE and the test configuration in the Debugger Window and the program cannot be stopped. This is indicated by the symbol.

If debug mode is active, you can set a breakpoint to the left of the line number with a mouse click. The breakpoint is red in network nodes and orange in test modules and test configurations.valid for CANoe DE, CANoe:lite DE

In test modulesvalid for CANoe DE, CANoe:lite DE and test configurations you can select for a breakpoint if the test modulevalid for CANoe DE, CANoe:lite DE or test configuration alone or also the simulation should be stopped. According to the selection the breakpoint is displayed red when stopping the whole simulation and orange when stopping only the test modulevalid for CANoe DE, CANoe:lite DE or test configuration. Via the context menu command Break Test/Simulation you can toggle between these functionalities.

If a breakpoint cannot be reached, it is marked as not active by an open circle .
Examples are

  • Empty lines
  • Comments
  • Variable definitions
  • Function definitions

To debug you must start the simulation.

Break

When a program reaches an active breakpoint during its execution, it is stopped. In doing so, the command in the line where the breakpoint was set has not yet been executed. At the same time, the values of all variables are updated and displayed in the Variables area. Variables that are valid at the current breakpoint are black, and all others are grayed. Examples of grayed variables are local variables that can only be used in other sub-functions. The values of simple variables are shown in a tooltip when you hover the mouse over the variable.

For breakpoints in test modulesvalid for CANoe DE, CANoe:lite DE or test configurations the program alone or also the simulation can be stopped, while in network nodes the entire CANoe simulation is stopped.

Breakpoints are also active if the Debugger Window is not currently open. The window is opened automatically in this case after the program is stopped.

The document view automatically switches to the file and line in the program in which an active breakpoint is reached. The command to be executed next is highlighted in yellow.

The CAPL function halt stops both the program as well as the simulation. The Debugger Window opens and the halt() command is displayed. The function thus creates a breakpoint while simultaneously interrupting the measurement in simulated mode (also for test modules and test configurations)valid for CANoe DE, CANoe:lite DE. You can use the single step functions to analyze the subsequent steps or continue the measurement with Start.

Continue

Note

If you want to use the <F5>, <F10> and <F11> keyboard commands described below while the measurement is in progress, you must observe the following points:

  • Activate the keyboard commands. Therefor you have two possibilities:
  • In the status bar of your CANoe DE product click the button with the keyboard icon , or
  • Use the key combination <Ctrl> + <K>.
  • Put the focus in the Debug Window:
  • Wait until the breakpoint is displayed, then click into the Debug Window.

You can use <F5>, the Continue context menu command or the icon in the toolbar to continue the simulation of your CANoe DE product until the next breakpoint is reached.

You can use <F10>, the Step Over context menu command or the icon in the toolbar to continue the program and, if applicable, the simulation until the next command in the current function is reached. If the end of a function is reached, the program will not be stopped until the next breakpoint.

Step Into (<F11>/) is different from Step Over in that the debugger also changes to sub-functions if they are present in the source code. This also applies to functions in include files. The debugger cannot change to functions of C/C++ DLLs or to functions integrated in the CANoe DE product. In this case, the function of Step Into is identical to that of Step Over.

In test modulesvalid for CANoe DE, CANoe:lite DE and test configurations, it is possible to change to .NET test libraries that have been created with your CANoe DE product. This requires the associated PDB file to be present in parallel with the .NET test library. If in this case the configuration in the CANoe DE product is executed on a computer on which the .NET test library was not created, the source files must be copied onto this computer into the same path as on the computer on that the test library was created. Otherwise, the source files cannot be found for displaying in the Debugger Window because the path to the source files is stored in the PDB files.

Test modules and testvalid for CANoe DE, CANoe:lite DE configurations can be started with Step Into or Step Over. The execution is stopped before the first command.

If the debugger has stopped the entire simulation, a measurement start () also resumes the simulation and the program.
