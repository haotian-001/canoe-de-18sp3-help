How to Create a New Configuration With SOME/IP Simulation Nodes and Application Objects

Create SOME/IP Application Object

  • a database is assigned to the configuration (e.g. AUTOSAR Classic or AUTOSAR Adaptive)
  • the Use application layer objects option is activated in the Options dialog.

When activated, application objects are created for all SOME/IP services and AUTOSAR PDUs described in the assigned database.

Note

If SOME/IP application objects are used, SOME/IP service signals can only be accessed read-only. Write access to a SOME/IP service signal then results in a runtime error. This is displayed in the Write Window when write access takes place. If no application objects are used, SOME/IP service signals can be accessed both read and write.

In contrast to SOME/IP service signals or PDUs, SOME/IP application objects are not transmitted via the network. Their scope of validity is limited to the application itself (e.g. CAPL code or .NET code). If a simulation node is switched off, the application objects lose their validity.

SOME/IP service signals are transmitted via the network. They can therefore also be used in pure measurement scenarios, i.e. even if there are no simulation nodes.

If the value of a SOME/IP application object is set, the associated service signal is updated the next time the SOME/IP message is sent. The time is linked to the transmission behavior of the SOME/IP message and is determined by the AUTOSAR database.

SOME/IP application objects are displayed in the Symbol Explorer on the Application Layer Objects page. They are sorted there according to network participants.

Use Cases

Example: SOME/IP Event

The AUTOSAR database defines a Vehicle service that is offered by the BCM participant and consumed by the IC participant. The service contains the GeneralDeviceState event, which contains the two parameters Speed and GearMode. The created Speed and GearMode application objects are displayed in the Symbol Explorer for both the BCM and IC participants (see image below).

The application objects Speed and GearMode of the participant BCM can be changed as it offers the service. If the GearDriveState event is triggered (left image), the current values are transmitted via the associated SOME/IP service signals Speed and GearMode (center image).

After receiving the SOME/IP message, the transmitted values can be queried via the associated application objects. The Speed and GearMode application objects of subscriber IC can only be accessed in read-only mode (image on the right).

Example: SOME/IP Method

The ATUTOSAR database defines a Lightning service that is offered by participant BCM and consumed by participant HPCM. The service contains the CommingHome method. The method has an input parameter Delay; the method ConfirmDelay returns the return value. The created application objects Delay and ConfirmDelay are displayed in the Symbol Explorer for both participant BCM and participant HPCM (see image below).

The application object Delay of the participant HPCM can be changed as it can call the method. If the CommingHome method is called by HPCM, the corresponding SOME/IP message (request) is sent (right-hand image). The value is now also included in the service signal Delay (middle image).

Once the SOME/IP message has been received, the method parameter can also be queried via the application object Delay of subscriber BCM (left-hand image). After the CommingHome method has been executed, the result is communicated via a corresponding SOME/IP message (response) and can be queried via the ConfirmDelay application object of the HBCM subscriber (right-hand image). The return value is also stored in the ConfirmDelay application object of subscriber BCM (left image) and can also be queried via the ConfirmDelay service signal (middle image).

Supported Databases

Your CANoe DE product supports both AUTOSAR Classic and AUTOSAR Adaptive databases.

Up to Version 17

  • AUTOSAR Classic databases can be used both in the Simulation Setup and in the Communication Setup.
  • AUTOSAR Adaptive databases can only be used in the Communication Setup. A Binding block is required to access the physical network.

Starting with Version 18

AUTOSAR Classic and AUTOSAR Adaptive databases can preferably be integrated in the simulation setup but also in the Communication Setup.

If the participants are simulated in the Simulation Setup, the previously available interaction layers can also be used with AUTOSAR Adaptive. A binding block is not required. Application objects can be used if you have activated the Use application layer objects option in the Options dialog.

SOME/IP Application Objects on Panels

You can assign SOME/IP application objects to both display and control elements.

Note

SOME/IP application objects can only be used in panels if the associated subscriber is also simulated in your CANoe DE product. If a node is deactivated in the Simulation Setup, controls that are linked to application objects of the subscriber can no longer be used.

Visualization of Signal Values

In contrast to application objects, SOME/IP service signals are also available if the simulation node has been deactivated. Therefore, only use service signals for panels to visualize signal values.

Node Control

If, for example, you want to call a method via a panel and enter the parameters associated with the method, you cannot use SOME/IP service signals, as these are read-only when using application objects. In this case, you must assign the application objects of the method parameters to the corresponding control elements. If the subscriber belonging to the panel is deactivated, you can no longer use the panel, as the method is called by the real subscriber.

Qualification of SOME/IP Application Objects

You can access SOME/IP application objects using the notification $ available in CAPL. The qualifier used for this is composed as follows:

[application object].[path]

Application Object

The application object can be addressed via the following qualifier:

[Cluster]::[ECU Name]::[Service Instance]

Cluster

Cluster name from the ARXML description. The cluster name can be omitted if the application object can be uniquely addressed (only one cluster available or only available once in the case of several databases).

ECU Name

Node name from the Simulation Setup. If the application object is unique across the network, the node name does not need to be specified.

Service Instance

Name of the service instance from the database. It is displayed in the Symbol Explorer on the Application layer objects page.

Path

The path itself is formed as follows for events, fields and methods:

Set an Event

[Event].[Value]

Event

Name of the event from the database. The event is displayed in the Events group of the corresponding node in the Symbol Explorer.

Value

Name of the value from the database. The value is displayed in the Symbol Explorer below the associated event.

Example CAPL

Sets the Speed value of the GeneralDriveState event to 1. The event notification is then sent.

$VehicleNetwork::BCM::VehiclePSI1.GeneralDriveState.Speed.phys = 1;

Method Call

Method calls are initiated by the consumer and must therefore be triggered by application objects of the consumer.

[Method].CallAsync(Arg1, Arg2 …)

Method

Name of the method from the database. It is displayed in the Methods group of the corresponding node in the Symbol Explorer.

ArgX

Value of the method argument. The arguments are displayed in the Symbol Explorer below the associated method. A method can have several arguments.

Example CAPL

Calls the CommingHome method with the value 1 in the argument. The method only offers one argument; if there are several arguments, they are separated by a comma.

VehicleNetwork::HPCM::LightingCSI1.CommingHome.CallAsync(1);

Access to Method Return Values

Return values can be requested from both the provider and the consumer.

[Method].LastReturn.[ReturnValue]

Method

Name of the method from the database. It is displayed in the Methods group of the corresponding node in the Symbol Explorer.

ReturnValue

Name of the return value from the database. The value is displayed in the Symbol Explorer below the associated method. A method can have several return values.

Example CAPL

Returns the last return value of ConfirmDelay of the CommingHome method.

$VehicleNetwork::HPCM::LightingCSI1.CommingHome.LatestReturn.ConfirmDelay.phys;

Trigger a Field

A field is triggered in the provider as follows:

[Field]

Field

Name of the field from the database. The field is displayed in the Fields group of the corresponding node in the Symbol Explorer.

Example CAPL

Sets the value of the Mode field to 1. The Notification field is then sent.

$VehicleNetwork::EDRV::DrivingPSI1.Mode.phys = 1;

Call the Getter Method of a Field

Getter calls are initiated by the consumer and must therefore be triggered by application objects of the consumer.

[Field].Get.CallAsync()

Field

Name of the field from the database. The field is displayed in the Fields group of the corresponding node in the Symbol Explorer.

Example CAPL

Calls the getter method of the Mode field. A getter method does not contain any arguments.

VehicleNetwork::BCM::DrivingCSI1.Mode.Get.CallAsync();

Call the Setter Method of a Field

Setter calls are initiated by the consumer and must therefore be triggered by application objects of the consumer.

[Field].Set.CallAsync(Arg)

Field

Name of the field from the database. The field is displayed in the Fields group of the corresponding Symbol Explorer node.

Arg

Value of the setter argument. The argument is displayed in the Symbol Explorer below the associated field. A Setter field can only have one argument, but the argument can be structured.

Example CAPL

Calls the setter method of the Mode field with the value 2 in the argument.

VehicleNetwork::BCM::DrivingCSI1.Mode.Set.CallAsync(2);

Access to Setter/Getter Return Values

Return values can be requested from both the provider and the consumer.

[Field].Get.LastReturn.[ReturnValue]

Field

Name of the field from the database. It is displayed in the Fields group of the corresponding node in the Symbol Explorer.

ReturnValue

Name of the return value from the database. The value is displayed in the Symbol Explorer below the associated method.

Example CAPL

Returns the last return value of the getter method of the Mode field.

$VehicleNetwork::BCM::DrivingCSI1.Mode.Get.LatestReturn.Result.phys;

Using Application Objects in CAPL

Note

You can conveniently drag application objects in the CAPL Browser from the Symbol Explorer into the Document View. The complete qualifier of the application object is then automatically generated and displayed.

How Tos (Use Cases)

How to Create a New Configuration With SOME/IP Simulation Nodes and Application Objects

Create New Configuration

  1. Start your CANoe DE product

  2. On the start screen, switch to the New page.

  3. Under Available Templates, select the Ethernet (Application Objects) template and click on Create configuration from template.

  4. In the Create Configuration dialog, select a configuration name and a directory. Activate the Create directory for configuration option to automatically create a sub-directory with the name of the configuration.

  5. Confirm your entry with [OK] and a new, empty configuration is created with the settings of the selected template.

    		6.

    Note

    The setting for creating application objects is already activated in the Ethernet (Application Objects) template. If you use your own templates, you must activate this setting in the Options dialog.

Assign AUTOSAR Database

  1. Switch to the Configuration desktop.

  2. In the Simulation Setup under Networks|Ethernet Networks|Ethernet1 select Databases.

  3. In the context menu, click on Add... and select an AUTOSAR database in the Explorer.

Add Simulation Node

  1. In the Simulation Setup, select the Insert Network Node command in the context menu of the red/blue line. A new simulation node now appears in the Simulation Setup.

  2. Open the Node Configuration dialog via the context menu command Configuration... context menu command of the newly created simulation node.

  3. On the Common page, select the desired node from the assigned database in theNetwork node field; if there are no nodes in the selection field, the assigned database could not be read in correctly.

  4. In the Node configuration dialog, switch to the Components page to assign an Interaction Layer DLL to the simulation node.

  5. Click on [Add...] and navigate in the Explorer to the Exec64 directory of your product installation. Select the CANoeILNL_AUTOSAR_Eth.dll file here.

Repeat steps 9-13 until you have added all the desired participants.

Measurement

  1. Start the measurement.
    If you do not have Ethernet hardware available, you must select the simulated mode on the Home ribbon tab. If the database is correct and you have inserted all the necessary network participants in the Simulation Setup, you should now see service discovery messages in the Trace Window.

  2. Switch to the Simulation ribbon tab.

  3. Open the Application Panels of the participants via Application Panel.
    Here you can now trigger events or fields or initiate method calls.
    You can also control the simulated participants using suitable CAPL functions.
