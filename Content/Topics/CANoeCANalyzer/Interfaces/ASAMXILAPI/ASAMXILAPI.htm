ASAM XIL API

Ribbon: Environment Tab|Tool Couplings|XIL API/FDX

Valid for:  CANoe DE  •  CANoe:lite DE  •  CANoe4SW DE  •  CANoe4SW:lite DE

The ASAM XIL API is a standard that specifies software interfaces for access to test systems. This API allows different test systems to be programmed in a standardized manner. Furthermore, investments in test scripts remain protected even when a change is made from one test system to another.

The API separates the test automation tool (client) from the testbench (server). The client can program against this API and the server must implement the API. Vector supports both use cases.

XIL API Installation

The components of the XIL API consist of

  • the ASAM AE XIL API standard assemblies form ASAM e.V.
  • the XIL API Implementation for your CANoe DE product. This implementation is made available in the form of a .NET component by Vector Informatik GmbH. It contains all implementations of the XIL API ports currently supported by your CANoe DE product (e.g., the Model Access port).

With the execution of the Vector XIL API setup both components are installed.

Installing the XIL API

In order to use the XIL API implementation for your CANoe DE product, you need to install the required assemblies on your computer. To do so execute the setup: Start Setup

During installation the following file will be created:

C:\ProgramData\ASAM\XIL\Implementation\Vector.CANoe.imf

This file is required by the XIL API test bench. It is used to locate and instantiate the XIL API at runtime.

Test Bench Initialization

XIL API test benches are instantiated by calling ITestBench.CreateVendorSpecificTestbench.

To create the CANoe DE product test bench you have to use the following parameters:

  • vendorName: must be set to Vector
  • productName: can be either <product name>32 or <product name>64.

    This settings depends on what architecture your XIL API client application is using.

    It does not correspond to the CANoe DE product version being used.
  • productVersion: the version of the XIL API interface to use (2.1.0 or 2.2.0).

Backend Selection

Your CANoe DE product offers two different backends for its XIL API implementation: FDX and LightRPC. You can choose between these backends implicitly via the port configuration files. Whenever a FDX configuration block is found in the file, FDX is used as the backend. If you omit the FDX configuration block, LightRPC will be used by default.

LightRPC Backend

LightRPC is a proprietary network-based protocol. It allows for a fast data exchange between your CANoe DE product and a client application. The main benefit of the LightRPC backend is that requires much less configuration effort compared to the FDX backend. You do not need to specify the exchanged variables and signals beforehand. All information will be retrieved automatically during the runtime.

Using the LightRPC backend is recommended for most users. It is easy to set up and provides high data throughput and low latencies.

FDX Backend

CANoe FDX (Fast data eXchange) is an open network protocol for fast data exchange with your CANoe DE product. The main benefit of the FDX backend is that it provides even higher data throughput than the LightRPC backend. It produces less communication overhead and can thus read/write more values per second. Also the FDX backend supports advanced data types (e.g. maps) that are not available when using the LightRPC backend.

Using the FDX backend is recommended when you want to transfer huge amounts of data between your CANoe DE product and your XIL API client.

Client Use Case (XIL Binding)

General

There are two ways in which you can connect third-party testbenches via the XIL API interface.

  • For use cases that only use the MA port to read and write variables, there is a solution based on the so-called XIL Binding. This solution is described here.
  • The C# ASAM XIL standard assemblies can be used directly in .NET based application models, network nodes or via .NET code in test units. Here you can use all ports of the interface. You can find more information on this in the official ASAM XIL documentation, which also contains code examples for C#.

Requirements

Make sure that the following conditions are met:

  • The XIL API Server used supports ASAM XIL API version 2.1.0 and has been compiled for the .NET Framework.
  • The XIL API Server supports the ASAM XIL Test Bench API for the MA port.
  • The IMF file of the XIL API Server is available.
  • The ASAM XIL Standard Assembly Interfaces are installed either in the Global Assembly Cache or in the ASAM XIL standard directory on the client system.
  • The server-specific configuration file for the MA port is available.

Technical Concept

In order to implement convenient read and write access to testbench variables, the variables are modeled using abstract variables, so-called distributed objects (DO). The modeling is initially independent of the transmission channel, but you can set this via a so-called binding. The binding is set to XIL here. Modeling is carried out in the programming language-independent description language vCDL. Not only the testbench variables and the binding are described in the vCDL file, but also all XIL-specific settings, such as configuration files and manufacturer name. You can use the command line tool xil-vcdl-generator to read testbench variables and generate a suitable vCDL file.

vCDL Modeling

The vCDL file is the central element in XIL binding. As soon as you have created it and loaded it in the Communication Setup, you can stimulate testbench variables via DO members and read them out cyclically.

XIL-Client Modeling with Distributed Objects

Example

Work with virtual networks to encapsulate all shared settings and inherit them on to the individual interfaces.

version 2.0;
import module "XIL";
namespace RoomTemperatureControl
{
  [Binding="XIL"]
  [XILBinding::VendorName="XILTestBenchVendor"]
  [XILBinding::ProductName="XILTestbench"]
  [XILBinding::ProductVersion="1.2.3"]
  [XILBinding::MAPortConfigPath="C:/path/to/my/PortConfig.xml"]
  [XILBinding::SampleTimeMs=100]
  [XILBinding::MAPortName="MAPort"]
  [XILBinding::ImplementationManifestFolder="C:/ProgramData/ASAM/XIL/Implementation"]
  virtualnetwork MAPortNetwork;
  [VirtualNetwork=MAPortNetwork]
  interface IHeating
  {
    [XILBinding::VariableName="RoomTemperatureControl/Heating.State"]
    consumed data uint64 State;
  }
  IHeating Heating;
  [VirtualNetwork=MAPortNetwork]
  interface ISensor
  {
    [XILBinding::VariableName="RoomTemperatureControl/Sensor1.Temperature"]
    provided data double Temperature;
  }
  ISensor Sensor;
}

Generating the vCDL File via the xil-vcdl-generator

The command line tool xil-vcdl-generator uses the ASAM XIL MA port to read out all testbench variables and generate the corresponding vCDL file. All settings can be made in a configuration file. The configuration file has the extension .xil-vcdl.yaml.

Configuration File

Example of minimal configuration file (*.xil-vcdl.yaml)

The following example shows a simple configuration file in which all testbench variables are read and a corresponding vCDL file is generated.

version: "1.0.0"
testbench:
  vendor-name: " XILTestBenchVendor"
  product-name: " XILTestBench"
  product-version: "1.2.3"
ma-port:
  config-file-path: "path/to/my/PortConfig.xml"

variables (optional)

Can be used if only a subset of testbench variables is required and/or the structuring (namespace, interface, object) of the generated DOs is to be adapted. variables is a list of structures that contain key-value pairs. If no structures are defined, all testbench variables are processed.

You can use the filter key to select specific testbench variables that are to be filtered during generation. You can define what the structure or naming (namespace, DO, member) should look like for the DOs generated for this purpose. As soon as you have defined a filter, all testbench variables that do not correspond to the filter are skipped. The first matching filter and its mapping rules are always applied.

The mapping rule from testbench variables to DOs is carried out using the keys format, nested-namespaces and communication-direction, which always belong to a filter.

filter

All testbench variables

Regular expression (RegEx) to make a selection of testbench variables.

Optional

format

%m

A default namespace and default DO object are created and the complete testbench variable name is used as the member name.

Format string that determines the resulting DO structure.

  • <characters to be skipped>%<number of separators>n<separator>
    Namespace Name (optional)
  • <characters to be skipped>%<number of separators>n<separator>
    Object/interface name(optional)
  • <characters to be skipped>%m
    Member name (mandatory)

The separators can consist of one or more characters.

The namespace separator can only consist of one character, is optional and is not compatible with <number of separators>. Here is <separator>=<Namespace-separator>

Note

vCDL does not support special characters, these are removed by the generator.

Optional

namespaces

—

namespaces is a structure and contains the elements nested and separator.

  • nested: Defines whether nested namespaces should be created or not. The default value is true.
  • separator: If nested is set to true, a separator can also be specified to split the namespaces. If no separator is specified, the separator of the namespace format string is used.

Optional

communication-direction

provided

Defines the DO communication direction (consumed or provided), if the testbench variable is marked as readable and writable.

Example of a complex configuration file (*.xil-vcdl.yaml)

The following example shows a more complex configuration file in which only certain testbench variables are generated. In addition, some default values are overwritten.

version: 1.0.0
general-settings:
  root-namespace-name: "CustomNamespace"
  variable-sample-time-ms: 250
testbench:
  vendor-name: "XILTestBenchVendor"
  product-name: "XILTestBench"
  product-version : "1.2.3"
  implementation-manifest-folder : "."
ma-port :
  config-file-path : "MAPortConfig.xml"
  name: "MAPort123"
  runtime-config-file-path: "C:/dev/MAPortConfig.xml"
  force-config: true
variables:
  - filter: "Seat"
    format: "n/%o.%m"
  - filter: "Engine"
    nested-namespaces: false
    communication-direction: consumed
  - filter: "."
    format: "%n/%o.%m"
