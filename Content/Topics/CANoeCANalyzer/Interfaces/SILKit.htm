SIL Kit

 

Valid for:  CANoe DE  •  CANoe:lite DE  •  CANoe4SW DE  •  CANoe4SW:lite DE

The SIL Kit is a software library that connects the System Under Test (SUT) with simulators, test tools and models at the different levels of abstraction needed during the various development phases. It enables communication from the application layer down to the frame level. SIL Kit's free and open architecture and robust interfaces provide a standard for long-term, vendor-independent interoperability. SIL Kit enables tools to connect seamlessly, whether they reside on the same machine or are distributed across multiple real or virtual machines, or even in the cloud. The open source C++ implementation ensures compatibility with a wide range of operating systems, e.g. Windows, many Linux distributions and other UNIX derivatives such as QNX.

The goal of SIL Kit support in your CANoe DE product is to perform remaining bus simulations and ECU testing in virtual environments. With the help of the SIL Kit, virtual ECUs can be tested with your CANoe DE product in the same way as is possible with real ECUs on an HIL test bench.

Requirements

To be able to use the framework, an implementation of the SIL Kit must already be installed on the system. The SIL Kit is installed by default as part of the CANoe DE product installation and does not need to be installed separately. The runtime components of the SIL Kit can be found in the corresponding installation folder of the SIL Kit (SilKit.dll, sil-kit-registry.exe and sil-kit-system-controller.exe).

Note that the SIL Kit is used as part of the simulation by the CANoe DE product runtime kernel and that the runtime kernel can also run on a separate device depending on the setup (RT Setup). The runtime components of the SIL Kit must be installed on the system on which the CANoe DE product runtime kernel is executed.

The SIL Kit has a detailed documentation on how to use the framework and a description of the API. This documentation is not delivered with your CANoe DE product, but can be viewed in the open source publication.

Supported Services of the SIL Kit

Your CANoe DE product can use the following services of the SIL Kit:

  • Network access for the CAN / CAN FD bus system

  • Network access for the Ethernet network

  • Network access for the FlexRay network

  • Network access for the LIN network

  • Exchange of application data via PubSub and distributed objects

  • Distributed function calls via RPC and distributed objects

Settings in the CANoe DE product

To use the SIL Kit, you must first activate it in the Options dialog. There you also make the necessary settings, which are necessary to join a simulation of the SIL Kit.

In the Application Channel Mapping dialog, you specify individually for each application channel whether the SIL Kit should be used. Application channels that are assigned to a network interface are automatically simulated in the Simulated Bus working mode of your CANoe DE product by the network simulator built into your CANoe DE product.

The time coupling between the CANoe DE product and the other participants of a simulation is determined by the set working mode of the CANoe DE product.

Real Bus Working Mode

Select the Real Bus working mode if all participants of the simulation are to run in realtime. In this case, no further time synchronization of the various simulation tools is required by the SIL Kit. Since your CANoe DE product runs in realtime in this case, it can also communicate with real vehicle networks via network interfaces in parallel to the data exchange via the SIL Kit. This means that mixed scenarios can also be implemented, in which one part of the vehicle networks is real and the other part is only virtual in the SIL Kit.

In this case, the timestamps of events (e.g. the display of transmitted frames on a CAN network in the Trace Window) are not set by the SIL Kit, but your CANoe DE product sets the time stamps when taking over the events from the SIL Kit. If you couple two CANoe DE product via a virtual CAN network of the SIL Kit, you will see different time stamps for the same CAN frame in the Trace Windows of the two CANoe DE product instances.

Simulated Bus (Slave Mode) Working Mode

In the Simulated Bus (Slave Mode) working mode the temporal progress of the simulation is controlled via the SIL Kit. Since your CANoe DE product does not run in realtime in this case, a mixed scenario together with real existing vehicle networks is not possible.

In this case the CANoe DE product takes over the time stamps of events of the SIL Kit. Two coupled CANoe DE product instances therefore also display identical timestamps in their Trace Windows for the same event.

When using the SIL Kit, the animation factor for this working mode must be set to the value 1.0.

At the start of measurement, the CANoe DE product waits for the SIL Kit to start the simulation. As usual for SIL Kit simulations, the set of required participants must be defined for the start (e.g. via sil-kit-system-controller.exe). Further relevant setting for this mode can be defined in the SIL Kit Configuration.

Simulated Bus (Animated with Factor) Working Mode

Similar to the Real Bus working mode, the participants of a simulation in Simulated Bus (Slave Mode) mode are not temporally coupled via the SIL Kit. The timestamps of events are assigned by the CANoe DE product when they are taken over from the SIL Kit. Two coupled CANoe DE product instances therefore show different timestamps in their Trace Windows for the same event.

In contrast to the Real Bus working mode, a simulation can be interrupted at any time (e.g. via the ribbon or the COM interface). That is why this working mode can be useful for some debugging scenarios where, in addition to other simulation participants, the simulation in the CANoe DE product must also be stopped.

Since your CANoe DE product does not run in realtime in this case, a mixed scenario together with real existing vehicle networks is not possible.

When using the SIL Kit, the animation factor for this working mode must be set to the value 1.0.

Simulated Bus (As Fast as Possible) Working Mode

The Simulated Bus (As Fast as Possible) working mode is not supported in combination with the SIL Kit.

Network Support

CAN

Each CAN channel in your CANoe DE product is connected to the corresponding CAN network of the SIL Kit via a virtual CAN controller of the SIL Kit. The SIL Kit CAN controller thereby connects to the SIL Kit network that has the corresponding name of the network connected to the channel.

 

Using the SIL Kit for virtual CAN networks is similar to working on a real CAN network. Here, several real ECUs, each with its own CAN controller, are connected to the CAN bus. The runtime kernel of your CANoe DE product is also connected to the CAN bus via a network interface (e.g. VN1630) with its own CAN controller. All simulated network nodes within the CANoe DE product that are assigned to a CAN channel and together represent a remaining bus simulation for this channel communicate with the CAN bus via a single CAN controller of the network interface.

Ethernet (channel-based access)

If you use Ethernet networks with channel-based access, the situation is similar to that for the CAN bus. The remaining bus simulation in your CANoe DE product for an Ethernet channel communicates with the network via a single Ethernet controller. In a simulation with SIL Kit, the network with real Ethernet controllers is replaced by a simulated network with simulated Ethernet controllers in the SIL Kit.

Ethernet (network-based access)

Each Ethernet port of an Ethernet network in your CANoe DE product is connected to the corresponding Ethernet network of the SIL Kit via a virtual Ethernet controller of the SIL Kit. The SIL Kit Ethernet controller thereby connects to the SIL Kit network that has the corresponding name of the network connected to the channel.

Since multiple ports can exist for a network within your CANoe DE product in the case of network-based access, multiple SIL Kit Ethernet controllers can also be created for the same Ethernet network in the SIL Kit. To be able to identify them uniquely, the SIL Kit controllers are named with a combination of network name and port name separated by a slash, i.e. <network name>/<port name>.

In configurations with a large number of ports in an Ethernet network, these individual connections of the ports to the SIL Kit lead to a large event load. By activating the option Optimize the required resources by connecting Simulation Ports with a simulated switch where possible in the Ethernet configuration, the performance of such configurations can be optimized. If this option is enabled, all ports of a network are connected to the SIL Kit via a single SIL KitEthernet controller. A virtual switch between this connection of the SIL Kit to the Ethernet network and the Ethernet ports within your CANoe DE product then forward the Ethernet packets to the individual ports. In this case, the corresponding SIL Kit controller is named according to the network.valid for CANoe DE,CANoe:lite DE.

Using the SIL Kit for virtual Ethernet networks is similar to working on a real Ethernet network. If you use Ethernet networks with network-based access, then there are multiple accesses to the Ethernet network for one Ethernet channel in your CANoe DE product. In this case, the network interface (e.g. VN5650) offers the functionality of an Ethernet network switch, through which the different Ethernet ports are connected to the Ethernet network.

FlexRay

The use of FlexRay networks is similar to CAN networks. The remaining bus simulation in your CANoe DE product for a FlexRay channel communicates with the network via a single FlexRay controller. In the real case this is the FlexRay controller from a connected network interface and in the virtual case with the SIL Kit it is a virtual FlexRay controller provided by the SIL Kit as a service.

Special Features

  • The network interfaces for FlexRay contain a hardware extension for the use of a second key slot. The FlexRay controller of the SIL Kit corresponds to a standard FlexRay controller with only one key slot. Therefore switch the key slot configuration to Manual and deactivate the second key slot.
  • The protocol support for PDUs available in the CANoe DE product is not available when using the SIL Kit.

LIN

The use of LIN networks is similar to CAN networks. The remaining bus simulation in your CANoe DE product for a LIN channel communicates with the network via a single LIN controller. In the real case this is the LIN controller from a connected network interface and in the virtual case with the SIL Kit it is a virtual LIN controller provided by the SIL Kit as a service.

If a node of the remaining bus simulation is configured as master node for a LIN channel, or if the channel was configured as master via the database configuration, the virtual LIN controller of the SIL Kit is configured as LIN master by your CANoe DE product.

Controller Names for Vehicle Networks

The SIL Kit requires a name for each network controller that a participant in a simulation wants to use. This name must be unique per participant in the context of a network type.

For most network types, your CANoe DE product uses the network name specified in the Simulation Setup as the controller name for logging on to the SIL Kit.

In the case of Ethernet with network-based access, however, the network name specified in the Simulation Setup is not sufficient, since several Ethernet controllers of the SIL Kit are typically used per channel. Therefore, a tuple is formed from the network name and the name of the simulation port. The two name components are separated by a slash. Independent of this controller name, the SIL Kit Ethernet controllers from your CANoe DE product connect to the SIL Kit network that has the name of the network of the corresponding controller. Thus, a Port1 on the Ethernet network Ethernet1 has the name Ethernet1/Port1 and connects to the network Ethernet1.

Application Data and RPC

Application Data

If the SIL Kit Binding is used in vCDL for data members or event members, they are mapped to the PubSub service of the SIL Kit. Provided members are mapped to DataPublisher and consumed members are mapped to DataSubscriber.

The name of the member is used as the Topic within the SIL Kit. The Media Type is set to application/vnd.vector.silkit.data; protocolVersion=1, because the de/serialization delivered with the SIL Kit is used.

Furthermore, vCDL specific information is encoded in the labels. This applies in the same way for DataPublisher and Subscriber.

RPC

If the SIL Kit binding is used in vCDL for method members, these are mapped to the RPC service of the SIL Kit. Provided members are mapped to RpcServer and consumed members are mapped to RpcClients.

The name of the member is used as the Functionname within the SIL Kit. The media type is set to application/vnd.vector.silkit.rpc; protocolVersion=1, since the de/serialization delivered with the SIL Kit is used.

Furthermore, vCDL specific information is encoded in the labels. This applies in the same way for RpcServer and RpcClients.
