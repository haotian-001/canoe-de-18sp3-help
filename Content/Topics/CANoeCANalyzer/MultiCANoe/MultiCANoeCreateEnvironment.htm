Multi CANoe Configuration [▲ back]

In a Multi CANoe cluster* All participants (CANoe instances) of a Multi CANoe configuration, you must distinguish between the coordinator and the agents.

Coordinator

As the central instance, the coordinator coordinates communication between the various agents and itself. For this, each agent registers with the coordinator independently

The coordinator monitors the cluster status. When all expected agents have registered, the cluster status changes from Incomplete to Connected and the measurement can be started.

You must specify a list of the instance names of all agents in the Options dialog.

Agents

For an agent, you must enter the IP address and port of the coordinator in the Options dialog. This allows the agent to independently establish a connection to the coordinator.

The settings are saved in the CANoe DE configuration (CFG). This means that they are retained even if a CANoe DE configuration with Multi CANoe is loaded on another PC.

All Multi CANoe participants (coordinator and agents) must be clearly distinguishable by their instance name. The coordinator uses the instance name CANoe by default, but this can be customized with the command line parameter /m, e.g. CANoe64.exe /m <coordinator name>. You can freely select the instance name for the agents. The instance name is relevant for the assignment of the system variables and for the channel assignment.

Transferring a Configuration From CANoe 17

Up to CANoe version 17, the settings for Multi CANoe were saved in the CAN.ini file and were therefore configuration-independent. In addition, all participants in a Multi CANoe cluster had to be specified. With version 18, the Multi CANoe configuration was changed. This section explains how you can transfer a configuration with CANoe version 17 (or earlier) to CANoe DE version 18.

Initial situation in CANoe version 17

Changes in CANoe DE Version 18

In the Options dialog, you must enter MultiCANoeS as the agent for the coordinator.

If you want to continue using the instance name MultiCANoeM for the coordinator, you can continue to start CANoe DE with CANoe64.exe /m MultiCANoeM.

For the agent, you must specify the IP address and port of the coordinator and set the instance name to MultiCANoeS. The instance name is used for the assignment of the system variables, if you also want to use it for the channel assignment, you must activate the Use instance name as application channel mapping name option.

Configuration of the VN Interfaces [▲ back]

You must configure the VN interfaces individually for each CANoe DE instance. Consider each CANoe DE instance as an independent CANoe DE and map the application channels with the hardware channels, as you normally would without Multi CANoe.

To operate several CANoe DE instances on one computer, you can activate the Use instance name as application channel mapping name option in the Options dialog. This automatically uses the CANoe DE instance name of the agent as the application name for the channel mapping. This also enables the configuration of different hardware channels to the CANoe DE instances if several CANoe DE instances are running on one computer.

Time Synchronization With the Vector Hardware Manager (Recommended)

You can configure the hardware connected to a computer in the Vector Hardware Manager.

With time synchronization via the Vector Hardware Manager, the time stamps are synchronized directly in the driver. You can choose between PTP and hardware synchronization. For PTP, it is possible to integrate an external PTP master. This allows the time to be synchronized across several computers. This is not possible with hardware synchronization.

Time Synchronization With CANoe DE

If several computers are connected to each other via hardware synchronization, you must configure the hardware synchronization in CANoe DE, as this use case is not supported by the Vector Hardware Manager.

Connect all VN interfaces with a corresponding SYNCcable. To enable CANoe DE to perform time synchronization independently, the connected VN interfaces must not have their own configuration for time synchronization assigned by the Vector Hardware Manager. You must therefore delete an existing configuration via the Vector Hardware Manager.

Time synchronization in CANoe DE is configured in the Network Hardware Configuration dialog.

An interface sends a pulse which is recognized by all connected interfaces. This pulse generator must be a VN interface that is connected to the Multi CANoe coordinator of the Multi CANoe environment.

The more interfaces have to be synchronized, the longer the transmission takes and the weaker the edge of the pulse becomes. With a Multi SYNCbox Active, you can amplify the input signal and forward it to up to 10 active outputs. This results in a constant delay between the input and output signals. This delay can be compensated for by measuring it and entering it in the CAN.ini file of the coordinator instance.

Measurement Start [▲ back]

In a Multi CANoe environment, the measurement can be started either by the coordinator instance or by any of the associated agent instances.

The measurement start is synchronized. As soon as you start the measurement, a request is sent to the coordinator and the coordinator sends a start command to all dependent CANoe DE instances. Each CANoe DE instance then starts its own configuration and reports the status back to the coordinator.

The measurement starts first in the coordinator and immediately afterwards in all agents. This means that the measurement starts with a slight delay in the agents. However, the time stamps are synchronized in all CANoe DE instances from the start and are therefore directly comparable with each other.

After the measurement start, the Multi CANoe cluster status is initially Starting. When the measurements have been fully started in all instances, the cluster status changes to Running and the system variable CANoe::MultiCANoeClusterRunning is set to 1.

While the Multi CANoe cluster status is Starting, it is possible that bus events cannot yet be processed by all CANoe DE instances. If, for example, a CAN frame is sent in an on start CAPL handler, this may not be received by another instance that is not yet fully initialized. If a bus communication that is to be processed by all instances is to take place directly after the measurement start, then this should only be sent when the cluster status is Running, e.g. via an on sysvar CANoe::MultiCANoeClusterRunning CAPL handler.

System variables for which the Multi CANoe write access is set can already be written in the cluster status Starting, as these system variables are subsequently synchronized between all instances when the cluster status changes to Running.

Data Exchange Between CANoe DE Instances [▲ back]

In principle, the configurations of the individual instances run independently of each other. To enable data exchange between the different instances of the Multi CANoe environment, system variables can be created whose value changes are passed on to the CANoe DE instances. For this purpose, an FDX connection is automatically established in the background via Ethernet.

Note

The system variables to be exchanged must be identical for all instances. It is therefore best to define all system variables in a CANoe DE configuration, save the system variables in a file (*.vsysvar) and import them into all participating CANoe DE instances.

In the System Variables Configuration dialog, you can define the Multi CANoe write access for each system variable and thus determine which CANoe DE instance may write the system variable. All other instances only have read access.

If you select the value All instances as write access, all CANoe DE instances can write to the variable and the values are synchronized with the other instances. If two CANoe DE instances write to a system variable at the same time, a warning about a possible data race is displayed in the Write Window.

If you have not explicitly defined write access, the variable is not synchronized in the Multi CANoe cluster and is treated as a local system variable.

For runtime reasons, you should only define a few system variables to be exchanged, as each update of a system variable leads to data exchange between all CANoe DE instances.

Note

Struct Definition with variable layout are not supported as system variables.

Messages in the CANoe DE Status Bar [▲ back]

If you have activated Multi CANoe in the Options dialog, the status bar displays the text Multi CANoe, the role of the current instance (coordinator or agent) and the state of the cluster:

Incomplete

Not all agents of the cluster have connected to the coordinator. The measurement cannot be started.

Invalid

At least one participant is invalid. Reasons are a wrong CANoe DE version of the participant or a wrong work mode. Multi CANoe can only be used with the Real Bus work mode. The measurement cannot be started.

Connected

All agents are connected to the coordinator, the measurement can be started.

Starting

One participant has initiated the start of the measurement and the remaining participants start the measurement.

Running

The measurement of the Multi CANoe cluster is in progress.

Stopping

One participant has initiated the stop of the measurement and the remaining participants stop the measurement.
