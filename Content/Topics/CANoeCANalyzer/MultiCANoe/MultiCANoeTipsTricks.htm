Time-Synchronous Execution of Actions on All Instances [▲ back]

An action can be initiated on a CANoe DE instance by changing the value of a common system variable. The transfer of the changed value can serve as a trigger for a separate action on all other CANoe DE instances. The execution will be very close in time.

However, the transfer of new values of system variables does not take place without delay. The time at which the instance with write access sets a new value is always earlier than the time at which the other instances see the update. You can make this visible in the Trace Window by comparing the timestamp of the update of the system variables on the different computers. The delay is normally less than 1 ms.

If you need to trigger a simultaneous action on multiple instances of CANoe DE, you can schedule this event in advance. Since the simulation time is the same on all instances, use a system variable to transmit a time in the future when you want to trigger an action. Use a second system variable to pass on the desired action to the other CANoe DE instances. Set up a timer in CAPL with the difference between the desired time in the future and the current simulation time, which you have determined using the CAPL function timeNowNS(). In this way, you can eliminate the transmission delay of the system variable values.

Conflict-Free Channel Assignment [▲ back]

If you want to use a Multi CANoe environment with, for example, two computers to examine and record a total of four CAN channels, then two channels could be assigned to each CANoe DE instance. As the CANoe DE configurations are independent of each other, you can use channel numbers 1 and 2 in both CANoe DE configurations. In both cases, bus events for CAN1 and CAN2 would then be recorded in the logging files. Make sure that it is clear from the name of the logging file whether the bus events were recorded by instance 1 or 2, as four different CAN networks are actually to be examined.

The subsequent data interpretation can be simplified as follows:

  • Create all four CAN networks in both instances.
  • Then switch off channels 3 and 4 in the first CANoe DE instance (set network to inactive).
  • Then switch off channels 1 and 2 in the second CANoe DE instance (set network to inactive).

In the logging data, each CAN network is stored with a unique number (CAN1 to CAN4) to avoid confusion during subsequent analysis.

This principle can only be used as long as no more channels are required than CANoe DE provides for the respective network type.

If you want to view logging files from different CANoe DE instances with the same channel numbers in the offline analysis, you can change the channel numbers in individual logging files for playback. To do this, open the Offline Mode Configuration dialog, Channel Mapping tab. Here you can specify the channel number for playback for each logging file.

Measurement Does Not Start [▲ back]

If the measurement of a Multi CANoe environment does not start, it is usually due to one of the following reasons:

  • A firewall is preventing communication.
  • The cabling for hardware synchronization is faulty.

If an error has occurred, it may have occurred in any CANoe DE instance. The error messages of an instance or of open pop-up windows are written to the Write Window of all participating instances. Further information on the error that has occurred can be found in the Write Window of the affected instance.

Check whether you can reach the IP addresses of all instances from the computer with the coordinator instance, e.g. by pinging the address via the command line. If this does not work, correct the Windows IP settings or check whether the firewall is preventing the connection.

Check the channel assignment in the Network Hardware Configuration and the cabling of the SYNCcable carefully.

If this does not help, it is advisable to set up a very simple Multi CANoe environment. Start with only 2 computers, each with only one network in the simulation setup and only one VN interface. If this works, expand your setup bit by bit. As soon as an error occurs, you know that the last change has to do with the problem.
