Asynchronous Evaluation

 

Valid for:  CANoe DE  •  CANoe4SW DE

If you have difficulties due to a high processor load in the simulation, you can increase performance by switching on the asynchronous evaluation of background monitoring. The calculations for the constraints and conditions are then executed asynchronously a little later. This does not change the reporting and, if applicable, the verdict of the tests.

The following images illustrate how asynchronous evaluation works in contrast to synchronous evaluation:

Synchronous evaluation of background monitoring

Asynchronous evaluation of background monitoring

In asynchronous evaluation, the constraints and conditions are calculated in a downstream step using the measured data and the result is written to the test report, but no simulation or stimulation takes place here. The asynchronous evaluation of the calculations frees up more computing capacity for the simulation and multi-core processors are better utilized.

You can activate asynchronous evaluation for all test units in the Options dialog or only for a specific test configuration in its execution options.

Asynchronous evaluation is not used for test modules.

CAPL Programming

In most cases, you will not have to change anything in your programs if you use asynchronous evaluation. However, there is an exception if you want to query the status of a check using the ChkQuery_... functions. Without asynchronous evaluation, these functions can be called synchronously. However, if asynchronous evaluation is activated, the data must also be retrieved asynchronously and you must wait for the result.

You can find out in the program whether asynchronous evaluation is used by calling the function testIsAsynchronousCheckEvaluationEnabled.

Example

stack long result;
stack dword checkId;
checkId = ChkStart_MsgRelCycleTimeViolation(VehicleMotion, 0.9, 1.1);
// ... execute test sequence

// Query whether asynchronous evaluation is used. Note that this is only for
// demonstration purposes in the example. You can use TestWaitForCheckQuery and
// the _AsyncResult functions also if asynchronous evaluation is not enabled; then
// the wait call will simply return immediately.
if (!TestIsAsynchronousCheckEvaluationEnabled ()) {
  // without asynchronous evaluation: synchronous query of check properties
  result = ChkQuery_NumEvents(checkId);
}
else {
  // with asynchronous evaluation:
  // Trigger asynchronous query of check properties and wait
  // until the data has arrived
  TestWaitForCheckQuery(checkId, 1000);
  // Retrieve the data
  result = ChkQuery_NumEvents_AsyncResult(checkId);
}

A further code adjustment may be necessary if you set up a check on the value of a system variable or a member of a distributed object (DO). If you have set the value directly beforehand without an intermediate wait point, this value change has not yet arrived in the asynchronous evaluation when the check is started, because such value changes are always processed at the wait points first. The check could therefore fail directly in the asynchronous evaluation. To avoid this, you must call the function testWaitForValueChangePropagation before starting the check.

Example

stack dword checkId;
@sysvar::SomeNamespace::MySysVar = 7;
TestWaitForValueChangePropagation();
checkId = ChkCreate_SignalValueChange(sysvar::SomeNamespace::MySysVar);
TestAddCondition(checkId);
ChkControl_Start(checkId);
