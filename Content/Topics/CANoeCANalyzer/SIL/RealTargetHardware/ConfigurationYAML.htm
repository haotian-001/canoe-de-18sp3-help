YAML Hardware Debugger Configuration

 

Valid for:  CANoe DE  •  CANoe:lite DE  •  CANoe4SW DE  •  CANoe4SW:lite DE

Overview

To specify which debuggers you want to connect to your CANoe DE product, you use a YAML configuration file. The debugger type must be selected for the debugger-specific settings to be taken into account. The debugger manufacturers supported by your CANoe DE product offer the option of connecting to debugger-specific software via TCP/IP. For this purpose, the IP (node) and the port for the respective debugger must be specified.

Top-level Settings

Settings

Mandatory (m) /
Optional (o)

Description

Possible Values

Minimum Version

version

m

With this setting, you can control which options are available.

"1.0", "1.1"

"1.0"

default-debugger

o

This setting enables to set one of the configured debuggers from the debuggers list as the default. This debugger is always used for communication if a distributed object is not assigned to an explicit debugger (vCDL).

One debugger from the debuggers list.

"1.0"

debuggers

m

List of debugger configurations that are to be connected to your CANoe DE product. The identifier of the debugger can be freely chosen and corresponds to a YAML node (see Example). The settings described in the following sections apply to a specific debugger.

List of debugger configurations. A configuration corresponds to a YAML node with a freely chosen name, for example MyGDBDebugger.

"1.0"

Generic Debugger Settings

Settings

Mandatory (m) /
Optional (o)

Description

Possible Values

Minimum Version

type

m

Type of debugger. Various specific settings are possible depending on the type.

gdb, isystem, lauterbach

"1.0"

node

m

IP address of the debugger remote API.

localhost, ip

"1.0"

port

m

Port of the debugger remote API.

Any port number

"1.0"

reset

o

If active, a soft reset is started via the debugger API at the start of measurement. Whether a reset is possible depends on the respective target.

true, false (default)

"1.0"

run

o

If true, the execution of the program is started at the start of measurement.

It is not possible to set run to false with the debugger type GDB. Since false is the default value, run must always be set active in case of GDB.

true, false (default)

"1.0"

auto-connect

o

If true, your CANoe DE product connects directly to the debugger at the start of measurement. If deactivated, the connection can be established later via the IDebugger interface.

true (default), false

"1.0"

Debugger-specific Settings

Debugger-specific settings are done in a node with the name of the debugger type (isystem, lauterbach or gdb) on the same level as the generic debugger settings, see Example.

GDB

Settings

Mandatory (m) /
Optional (o)

Description

Possible Values

Minimum Version

path-to-gdb-client

m

Path to the GDB client. This is usually supplied with the compiler toolset.

Example: C:\Program Files (x86)\Arm GNU Toolchain arm-none-eabi\12.3 rel1\bin\arm-none-eabi-gdb.exe

Absolute path or path relative to the YAML configuration file.

"1.1"

path-to-symbol-file

m

Path to the binary running on the target (for example .elf). This must match the binary running on the target.

Absolute path or path relative to the YAML configuration file.

"1.1"

disable-winidea-polling

o

Deactivates the regular value update in the winIDEA GUI. Leads to lower jitter during communication when deactivated. Interactive work during communication with your CANoe DE product is then not possible in winIDEA.

"1.0"

Example

Example: YAML Debugger-Configuration (one Debugger in the Test Setup)

version: "1.1"

default-debugger: debugger3

debuggers:
  debugger1:
    type: lauterbach
    node: localhost
    port: 3000
    reset: false
    run: false
    lauterbach:
      pack-len: 1024

  debugger2:
    type: isystem
    node: localhost
    port: 5315
    auto-connect: false

  debugger3:
    type: gdb
    node: localhost
    port: 4444
    reset: false
    run: true
    gdb:
      path-to-gdb-client: "C:/arm-toolchain/bin/arm-none-eabi-gdb.exe"
      path-to-symbol-file: "C:/SomeSoftware/bin/App.elf"
