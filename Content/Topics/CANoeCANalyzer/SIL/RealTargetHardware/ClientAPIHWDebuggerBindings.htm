Client API of the Hardware Debugger Bindings

 

Valid for:  CANoe DE  •  CANoe:lite DE  •  CANoe4SW DE  •  CANoe4SW:lite DE

Introduction

The connection between your CANoe DE product and the SIL Adapter via a debugger is automatically established by default when the measurement starts and disconnected when the measurement ends. Using the client API of the hardware debugger binding, you have the option of controlling this behavior during the measurement and adapting it to your test procedure.

The client API is accessed via special distributed objects. To obtain these, you must create distributed objects with the predefined interface HwDebuggerBinding::IDebugger from the HwDebugger module and assign each one to a debugger from your YAML configuration. The assignment is made with the attribute HwDebuggerBinding::DebuggerName. The following example shows how such a distributed object can be created using a vCDL file:

vCDL Example: Creation of an IDebugger object

version 2.0;

import module "HwDebugger";

namespace MyNamespace
{
  // The binding for IDebugger objects is "HwDebugger" by default
  [HwDebuggerBinding::DebuggerName="MyDebugger"]
  HwDebuggerBinding::IDebugger MyDebugger;
}

Only one IDebugger object can be assigned to each debugger from the YAML configuration. If the DebuggerName attribute is not set, the object is automatically assigned to the default debugger of the YAML configuration. When accessing the object, e.g. by calling a method, only the assigned debugger and its connection are affected.

The data and methods of IDebugger objects are accessed exclusively from a CANoe DE product. This access can be done as usual via the means offered by your CANoe DE product, e.g. programmatically (CAPL, C#, Python) from application models and tests, or from panels. The IDebugger objects have no impact on the generation of the SIL Adapter and its interface.

In the following sections, you find a description of the data and method members and their data types, which are available to you at the IDebugger interface. This is followed by examples of their use.

Data Members

The following data members allow read access (consumed) to a value.

Name

State

Direction of Communication

consumed

Data Type

HwDebuggerBinding::DebuggerState

Default Value

HwDebuggerBinding::Offline

Description

Describes the state of the IDebugger object and the connection between the CANoe DE product and the SIL Adapter on the target system. The following method members can only be called if the object is in the correct state.

Methoden Members

The following method members can be called (consumed) by a CANoe DE product.

Method Name

HwDebugger_Connect

Direction of Communication

consumed

Description

Connects a CANoe DE product to the SIL Adapter on the target system via the referenced hardware debugger. Executed automatically at the start of measurement if the auto-connect setting in the YAML configuration of the debugger is set to true. The IDebugger object must be in the Offline state. After successful execution, it is in the Online state.

Parameter

Name

Data Type

Description

reset

Boolean

Performs a soft reset of the target system. Must be supported by the target system.

Return Values

Name

Data Type

Description

—

HwDebuggerBinding::ConnectResult

Describes the result of the connection setup.

Method Name

HwDebugger_Disconnect

Direction of Communication

consumed

Description

Disconnects the connection between the CANoe DE product and the SIL Adapter on the target system. Executed automatically when measurement is stopped. The IDebugger object must be in the Online state. After execution, it is in the Offline state.

Return Value

Data Type

Description

HwDebuggerBinding::DisconnectResult

Describes the result of the disconnection.

Name

Enumeration

Description

Describes the status of the IDebugger object and the connection between the CANoe DE product and the SIL Adapter on the target system.

Enumeration Values

Name

Description

Online

There is a connection and communication is possible.

The connection is in an error state. An attempt is automatically made to disconnect the connection and, regardless of the success, to switch to Offline state.

Examples

General instructions for programming with distributed objects and further examples for CAPL, C#, and Python can be found under Programming with the Communication Concept.

The following vCDL example is the basis for the following programming examples.

vCDL Example: Create IDebugger Object

version 2.0;

import module "HwDebugger";

namespace Namespace
{
  [HwDebuggerBinding::DebuggerName="MyDebugger"]
  HwDebuggerBinding::IDebugger Debugger;
}

The following example shows a manual connection setup at the start of a test and a manual disconnection at the end of the test. The connection setup performs a soft reset of the target system. Please note that the test functionality (testWaitForAnswer, testStepFail, testStepPass) is only available within a test.

CAPL Example: Connection and Disconnection

export testcase Test()
{
  callcontext Namespace::Debugger.HwDebugger_Connect ccConnect;
  callcontext Namespace::Debugger.HwDebugger_Disconnect ccDisconnect;
  long reset = 1;

  ccConnect = Namespace::Debugger.HwDebugger_Connect.CallAsync(reset);

  if (testWaitForAnswer(ccConnect, 10000) <= 0)
  {
    testStepFail("Connect", "Timeout");
  }
  else if (ccConnect.Result < HwDebuggerBinding::ConnectResult::Success)
  {
    testStepFail("Connect", "Failed");
  }
  else
  {
    testStepPass("Connect", "Success");
  }

  // Test communication / implementation

  ccDisconnect = Namespace::Debugger.HwDebugger_Disconnect.CallAsync();

  if (testWaitForAnswer(ccDisconnect, 10000) <= 0)
  {
    testStepFail("Disconnect", "Timeout");
  }
  else if (ccDisconnect.Result < HwDebuggerBinding::DisconnectResult::Success)
  {
    testStepFail("Disconnect", "Failed");
  }
  else
  {
    testStepPass("Disconnect", "Success");
  }
}
