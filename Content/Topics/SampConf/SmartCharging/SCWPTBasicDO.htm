WPTBasicDO

 

Valid for:  CANoe DE

Click on the blue link to load the sample configuration in your CANoe DE product.

  • WPTBasicDO – WPTBasicDO.cfg
    Demonstrates wireless communication between an electric vehicle (EV) and the charging station (EVSE) in accordance with ISO 15118-20 WPT.

Introduction

The configuration represents the most important features of the Option SmartCharging for WPT by connecting a vehicle simulation with a charging station simulation. You will find the following application models in the Communication Setup:

  • EVSEsimDO.can: Supplies data for the charging station simulation

  • EVsimDO.can: Supplies data for the vehicle simulation

  • Monitor.can: Helps to display the data

Desktops

This configuration contains the following desktops:

Simulation V2G Test Configuration

Simulation [▲ back]

On the desktop simulation, you can observe the communication of a single vehicle with the charging station. The behavior of the participants can be influenced via the corresponding panels. To start the communication, simply click on the Connect vehicle button. The simulation runs automatically until the end of the communication.

In the Trace window, the smart charging messages (SDP, V2G) are displayed in decoded form with complete data interpretation.

Note

When the battery is full, the vehicle automatically stops the session.

EV

This panel serves as a control and display element for the vehicle simulation module. The following controls are available:

Operator Control

Description

Connector button

Connects the vehicle to the charging station over WLAN or disconnects it again. As soon as the button is clicked, the SCC protocol sequence begins.

Battery

Indicates the current charging state charge status of the battery. The battery status can be reset using the [Clear] button.

Link

Displays the link status between EV and EVSE.

Vehicle state

The current state within the (simplified) SCC state machine.

Reset / Pause

The vehicle simulation can be paused and reset using the two buttons. A restart is triggered by a double click on the [Connector button] button.

Stop Charging

End the protocol in the regular way, from the charge loop.

ReNegotiation

Initiates a re-negotiation process while the protocol is in the charge loop.

MeteringConfirmation

Initiates a MeteringConfirmation process while the protocol is in the charge loop (only possible in PnC mode).

Certificate installation

Initiates a certificate installation during the next protocol execution (only possible in PnC mode).

Schema

Use this switch to select the protocol version to be used for the connection. The protocol version is adopted as soon as the SupportedAppProtocol message is sent.

Dynamic / Scheduled

Sets the ControlMode that is requested by the vehicle.

Charge Loop Interval

Controls for changing the loading loop interval, which defines the interval of successive requests within the charge loop.

Tab WPT

Access to values that are relevant for the WPT charging process: the voltage values of the requests.

Tab Limits

Electrical limits that are used in various messages.

EVSE

Description

SessionID

Protocol

In this panel, you can follow the ISO 15118-20 protocol sequence and configure the general behavior. When messages are received, the current protocol status is highlighted in the sequence view and the return value of the charging station is displayed. For simplicity, not every displayed status corresponds to a single message.

The following controls are available:

Operator Control

Description

Protocol bar

Indicates the current state of the protocol and the response codes sent by the EVSE.

Beacon frame received

Indicates that a beacon frame was received.

Processing Time

If this option is activated, the charge point simulates a processing time for each message pair that contains the EVSEProcessing flag and causes a message loop. If this option is not activated, the simulation continues directly with the next message pair.

TLS

Activates the use of TLS the next time a connection is established. The configuration is carried out automatically over the supplied security profile.

PnC Active

Here you can switch between PnC and EIM mode. Activate the Active check box for PnC mode.

V2G Test [▲ back]

This desktop is a demonstration of the simulation modules ability to send configurable SCC messages at arbitrary times. For V2G communication, one or both participants can be operated in passive mode. The V2G protocol flow can then be controlled manually by sending arbitrary SCC messages from the panel. This is intended to resemble a test scenario where the active unit is the System Under Test (SUT) and the passive unit is the tester. The tester can check whether the SUTs responds by evaluating the callback functions.

Protocol

You can configure and start the test scenario in this panel.

Operator Control

Description

Controlled Entity

This defines the entity to be controlled by the user. For example, if the vehicle is under control, it goes directly into passive mode, while the charge point operates its state machine and responds to every request sent by the user.

Note

You can still send both requests and responses on call, regardless of which participant is under control. This shows to how the free sending function can be used in both active and passive mode.

Schema

Selected scheme for messages to be sent.

Apply

Similar to the Connector button, this connects both participants, but also applies the passive mode as configured.

TLS

If this option is activated, the connection is established with TLS encryption.

PnC Active

If selected, the AuthorizationRequest and AuthorizationResponse are filled accordingly.

CertificateInstallation

If the EVSE is controlled, this button is used to decide whether the EV sends a CertificateInstallation message.

SendFunctions

This panel is used for sending SCC messages. All available messages are displayed on the panel and can be sent at any time. The only restriction is that a TCP or TLS connection must exist for all messages with the exception of SECC Discovery. Therefore, the first message sent must always be the SECC Discovery request or response (depending on the controlled entity). In addition, the SupportedAppProtocol should be sent, as the usage schema is read directly from this message pair. It is also recommended to send the SessionSetup messages in order to build a valid session ID. Otherwise, the counterpart will not accept the messages sent.

Note

The parameters for the messages are all defined by CAPL. Take a look at the CAPL code to see how messages can be parameterized.
